# Jest中的匹配器

# 真假匹配器

# toBe

toBe表示完全相等，内容相同的两个对象，类似===，是不能通过toBe的，比如

test('测试', () => {
  expect({key1: 'value1'}).toBe({key1: 'value1'}) // 不通过
})

# toEqual

toEqual表示相等，类似==，上述情况是可以通过toEqual的

test('测试', () => {
  expect({key1: 'value1'}).toEqual({key1: 'value1'}) // 通过
})

# toBeNull、toBeUndefined、toBeDefined

toBeNull只匹配null值，toBeUndefined只匹配undefined，toBeDefined可以匹配除了undefined的所有值，包括0、null。

# toBeTruthy、toBeFalsy

truthy表示真值，只要是if语句中会转为true的均为truthy值，如[] {} 1 -1

# toBeCloseTo

对于比较浮点数相等，使用toBeCloseTo 而不是toEqual，因为你不希望测试取决于一个小小的舍入误差。

test('两个浮点数字相加', () => {
  const value = 0.1 + 0.2;
  //expect(value).toBe(0.3);           // 这句会报错，因为浮点数有舍入误差
  expect(value).toBeCloseTo(0.3); // 这句可以运行
});

# 使用async和await

和上面的Promise同理，jest可以使用await将Promise中的值取出。

test('test async await', async () => {
  await expect(fetchData2()).resolves.toMatchObject({
    data: {
      success: true
    }
  })    // 通过
})

test('test async await2', async() => {
  const res = await fetchData2()
  expect(res.data).toEqual({ success: true }) // 通过
})

# 测试接口是否存在

test('should 404', () => {
  expect.assertions(1) // 断言 
  return fetchData2().catch(err => {
    // expect(err.response.status).toEqual(404)
    expect(err.toString().includes(404)).toBe(true)
  })
})

test('should 404', () => {
  return expect(fetchData2()).rejects.toThrow() // expect this Promise to be rejected an throw exception.

# jest的钩子函数和分组

# 钩子函数

  • beforeAll在一个测试文件的所有测试用例执行之前执行的方法。
  • beforeEach在一个测试文件的每一个测试用例执行之前执行的方法。
  • afterEach在一个测试文件的每一个测试用例执行之后执行的方法
  • afterAll在一个测试文件的所有测试用例执行之后执行的方法

# mock简述

mock为模拟的意思，jest可以模拟函数的行为，包括调用、传参、返回、获取调用的实例

  cb(...args)

export function forEach(items, cb) {

# 模拟返回值

使用cb.mockReturnValue方法可以使创建的方法模拟返回某一个值。将上面的例子稍微改动一下，将runCallback改为return cb(..args)

// mock.test.js
test('测试回调方法是否执行', () => {
  const cb = jest.fn()
  cb.mockReturnValue('fisheep')
  runCallback(cb, '1', '2', '3')
  runCallback(cb, 'java', 'script')
  expect(runCallback(cb, 'visual', 'studio', 'code')).toBe('fisheep')
  console.dir(cb.mock);
})

# React的TDD测试

# 何为TDD

Test Driven Development，测试驱动开发，先写测试用例，这是所有测试用例均不能通过，随着代码功能的实现，测试用例会逐步通过，开发流程为：

  1. 编写测试用例
  2. 运行测试用例，不能通过
  3. 编写代码，使测试用例通过
  4. 优化代码
  5. 完成开发
