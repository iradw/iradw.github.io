# setState的水有多深

# setState到底是同步还是异步

之前我一直都以为setState是一个异步方法，经过一番探讨，我发现我错了，setState有时同步有时异步，没错就是这么强。

  1. setState本身的执行过程是同步的，只是因为在react的合成事件(onClick、onChange等)与钩子函数中执行顺序在更新之前，所以不能直接拿到更新后的值，形成了所谓的异步。也就是说setState在React库内部事件里是“异步”的，在原生事件中是同步的。很好理解：内部事件有自己的执行顺序，setState不听你的，听react的；而原生事件不归react管，所以是同步的。
  2. setInterval大坑：一个简单的需求，页面加载完后导航栏的透明度逐渐增加至1，刚开始的写法是这样的
  const [opacity, setOpacity] = useState(0.1)
  useEffect(() => {
  setInterval(() => {
    console.log('aaa')
    setOpacity(opacity+0.1)
  }, 1000)
  console.log(opacity)
  }, [])

但是发现，opacity只增加了一次，以后一直都是0.2，‘aaa’却不停的输出。这就给我一种错觉，不是说setState在setTimeout和setInterval中是同步的吗？怎么又成异步了呢？因为useEffect传入一个空数组，只在组件加载的时候执行useEffect的第一个参数，所以setInterval中拿到的opacity永远是0.1，必须改成如下写法，才能保证setInterval中的opacity是最新的。

  const [opacity, setOpacity] = useState(0.1)
  useEffect(() => {
    setInterval(() => {
        console.log('aaa')
        setOpacity(v => v+0.1)
    }, 1000)
    console.log(opacity)
  }, [])

setOpacity(v => v+1)这个写法是形参为上一次的state 返回更新后的state，所以每次都能拿到最新的state。
